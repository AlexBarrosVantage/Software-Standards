Avoid SQL queries on XXL tables that no index can support (ABAP)

Technology

ABAP

Unified Status

Not Unified

XXL Status

XXL

Critical Quality Rule

Yes

Weight in Parent Tech. Criterion

9

Rationale

SQL Queries that do not use table indexes are extremely dangerous for the application's performance. Indeed, each execution of the query will result in a full table scan which is extremely time consuming especially if the table contains an extremely large amount of data (XXL table).

Description

This metric retrieves the ABAP Artifacts containing at least one Open SQL query on an XXL table not using the XXL table's indexes. An Open SQL Query is using table's indexes if all the following conditions are true: - At least one index is defined for each table participating in the from list - Queries must reference left-most columns of the index key XXL tables are extremely large tables which contain a huge amount of data. The threshold that determines when a table is considered to contain a huge amount of data can be configured by the user.

Remediation

Try to use indexed columns in WHERE clauses. If it is not possible, then create a secondary index for these columns.
