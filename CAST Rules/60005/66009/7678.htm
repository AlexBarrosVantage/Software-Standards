Avoid using specific logging implementation

Technology

JEE

Unified Status

N/A

XXL Status

N/A

Critical Quality Rule

No

Weight in Parent Tech. Criterion

3

Rationale

Instead of being stick to a specific logging implementation, it is recommended to use a logging wrapper such as slf4j. It provides a Log interface that is intended to be both light-weight and an independent abstraction of other logging toolkits. It provides the middleware/tooling developer with a simple logging abstraction that allows the user (application developer) to plug in a specific logging implementation.

Description

Finds and reports all references to log4j API usage. The threshold parameter allows specifying other specific implementations package to avoid. All artifacts that reference a class, an interface, a method or a field that belongs to a class from the package org.apache.log4j will be reported.

Remediation

Use SLF4J http://www.slf4j.org/index.html instead and configure it to use your specific logging implementation.

Reference

http://www.infoq.com/news/2007/02/javaee-best-practices

Sample
import org.apache.log4j.*
Logger  logger = Logger.getLogger("com.cast"); // VIOLATION
Remediation sample
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

Logger log = LoggerFactory.getLogger("com.cast"); // FIXED
Output

This report lists all artifacts and fields that violate this rule. It provides the following information: Artifact Full Name and Field Full Name

Total

total of artifacts and fields that use org.apache.log4j, org.apache.commons.logging or other logging package specified in the threshold
